最近写 skill 写得比较多以后，我越来越觉得很多人会把它理解错。

最容易出现的写法，是把 skill 写成一段更长的提示词：你要专业、你要准确、你要遵守特定风格、你要一步一步思考。这样写短期看起来有用，因为它确实给模型增加了一些要求。但任务稍微变复杂一点，问题就出来了：它没有告诉 Agent 什么时候该触发，没有告诉它资料在哪里，没有把容易犯错的地方拦住，也没有规定输出前怎么验证。

这里说的 Agent，指的是那种能读取上下文、调用工具、编辑文件，并且围绕一个任务持续推进的 AI 助手。

真正有价值的 skill，更像是给 Agent 准备的一份工作现场。它里面应该有入口、流程、红线、资料、工具和复盘机制。用户提出任务以后，Agent 能知道自己该进入哪条路径，先查什么，哪些事情不能凭记忆猜，哪些步骤应该交给脚本，最后用什么动作确认结果没有跑偏。

我现在更关心的是：一份 skill 怎样才能真正改变 Agent 的行为。

## Skill 解决的是上下文问题

AI 模型本身已经知道很多通用知识。写 skill 时最浪费的做法，是把百科式背景塞进去。

比如写一个“某语言代码生成”的 skill，不需要从语言历史、设计哲学、生态概况讲起。Agent 真正需要的是这些东西：

- 当前语法和旧式语法有什么区别。
- 哪些写法看起来像其它语言，但在这门语言里不能用。
- API 的事实来源在哪里，是本地源码、官方文档、schema（结构化数据或协议字段定义），还是项目里的生成产物。
- 修改后应该跑什么命令验证。
- 出错时应该先判断语法错误、生成错误、构建错误，还是运行时错误。

这些信息不会让文章或文档显得宏大，但能直接改变 Agent 的行为。

写作类 skill 也是一样。告诉 Agent “写得有深度”没有太大意义。更有用的是告诉它：标题不要写营销风格；开头从真实问题进入；编号只在顺序有意义时使用；如果用了技术图，必须渲染检查文字有没有截断；不要把一句机械对照句当成文章主旨。
这些规则比“写得专业”更接近可执行指令。

所以 skill 的第一层判断很简单：这段内容会不会改变 Agent 的下一步动作。如果不会，它大概率不应该出现在 skill 里。

## description 决定 skill 会不会被用上

一个 skill 的正文再好，如果触发不准，效果也会很差。

支持 skill 机制的 Agent 通常会先看到 skill 的 `name` 和 `description`，也就是 `SKILL.md` 顶部 frontmatter 里的 YAML 元信息。正文只有在 skill 被触发后才会加载。所以 `description` 不能只写一句“用于处理某类任务”，它要把触发场景、任务对象和排除边界尽量写清楚。

比如一个写作类 skill，如果只写：

```yaml
description: Help write better articles.
```

这个描述太宽。用户让 Agent 写 README、产品介绍、接口文档、博客文章时，它都可能触发。更好的描述应该说清楚对象：

```yaml
description: Write or revise personal Chinese technical blog posts, including title selection, article structure, style cleanup, publishing metadata, and diagram checks. Do not use for README, product docs, API docs, or requirements unless explicitly requested.
```

代码类 skill 也一样。如果它只处理某个 UI 组件框架，就不要写成“前端开发 skill”；如果它只处理某个内部 DSL（领域专用语言），就要写明 `.dsl` 文件、语法、诊断、生成代码和构建验证这些触发词。
description 写窄一点不会削弱能力，它主要是在减少误触发。

我现在会用一个问题检查 description：只看这一段，Agent 能不能判断用户的新请求该不该加载它。判断不了，就说明 description 还不够具体。

## SKILL.md 负责流程和红线

`SKILL.md` 的正文不要写成完整知识库。

它更像操作规程：任务来了以后，先判断什么，读哪些资料，哪些规则绝对不能违反，输出前要做什么验证。

一个很小的骨架大概是这样：

```md
---
name: skill-name
description: Use when ...
---

# Skill Name

## 工作流

- 先判断任务类型。
- 按任务读取必要 reference。
- 输出前执行对应检查。

## 硬规则

- 不要编造本地不存在的 API。
- 不要跳过构建或渲染验证。

## 参考入口

- 语法规则：`references/syntax.md`
- 验证脚本：`scripts/validate.py`
```

这个骨架看起来普通，但已经把 skill 最重要的边界摆出来了：它什么时候触发，触发后怎么走，细节从哪里来，结果怎么确认。

很多 skill 写不好，是因为 `SKILL.md` 里塞了太多解释。比如一个文档处理 skill，把 Markdown、DOCX、PDF、图片排版、字体、分页、渲染检查全部写在主文件里。每次只改一个 Word 标题，也要加载一堆无关内容。更好的做法是：主文件写流程和分流条件，具体格式规则放到 reference 里。

## 硬规则要来自真实失败模式

好 skill 里的“不要”，作用是拦住模型最自然的错误。

Agent 很擅长迁移经验。它看到某个平台任务，可能会把另一个平台的依赖声明、布局模型或生命周期概念带进来；看到一门 DSL，可能会套用更常见语言的语法；看到写文章，可能会自然生成空泛时代背景和“深入探讨”式开头；看到技术图，可能会把很多长句塞进节点里，最后渲染出来文字被裁掉。

这些问题如果只靠“请准确”“请认真”，基本拦不住。skill 里要把它们写成具体规则：

- 使用任何框架 API 前，先查本地源码或官方文档。
- 不要把其它语言的语法迁移到当前 DSL。
- 不要生成项目里不存在的配置项。
- 技术图文字很多时使用 SVG、Mermaid 或 HTML/CSS，不要用不可控的生成式图片。
- 交付前必须实际渲染图片或构建项目。

硬规则最好来自真实返工。你发现 Agent 第三次犯同一个错时，就不要只在对话里提醒它了，应该把这条错路写回 skill。

## reference 放细节，script 放确定性

Skill 目录里最常见的资源是 `references/`、`scripts/` 和 `assets/`。

`references/` 放需要按需阅读的资料。比如写作风格画像、API 说明、语法速查、错误诊断、UI 质量规则、协议说明、数据库 schema。它们通常比较长，不适合每次都进上下文。主文件只需要写清楚：什么任务读哪份 reference。

`scripts/` 放重复且容易出错的动作。比如解析配置、搜索 API 索引、转换文件格式、生成缩略图、跑构建、还原错误行号、校验 frontmatter。凡是每次都要重新写命令、重新解析路径、重新处理一堆边界的动作，都可以考虑脚本化。

`assets/` 放输出时要复用的素材。比如文档模板、幻灯片模板、前端项目骨架、品牌图片、默认字体。很多 skill 用不到 assets，不需要为了目录完整硬塞。

这三个目录解决的是不同问题：

- reference 减少“Agent 不知道事实来源”。
- script 减少“Agent 临场写错操作步骤”。
- asset 减少“Agent 每次重新造模板”。

大资料尤其要谨慎。比如一个代码生成 skill 需要查几千个 API，不应该把所有 API 文档直接写进 `SKILL.md`。更合理的方式是做索引，让 Agent 先搜索，再按命中的 `file:line` 打开相关片段。这样既有事实依据，又不会把上下文窗口撑爆。

## 自由度要按任务风险来调

不同 skill 的约束力度不应该一样。

写作类 skill 可以保留比较高的自由度。标题、段落顺序、例子取舍和表达方式都可以根据材料调整。这里更适合写风格规则和质量检查，固定模板反而容易把文章做成流水线产物。

代码生成类 skill 要收紧很多。语法、API、继承关系、生命周期、配置项，这些东西写错就会导致编译失败或运行错误。这里需要更多硬规则，也需要明确事实来源：本地源码、官方文档、schema、生成文件，哪个优先级最高。

构建和部署类 skill 的自由度更低。命令、路径、环境变量、构建阶段、失败归因都很容易错。能脚本化就脚本化；不能脚本化，也要写清楚先检查什么、哪些参数不能猜、失败时保留哪条日志。

文档、表格、PDF、图片这类 skill 也类似。只要最终效果和布局有关，就不能只相信生成文件存在。应该要求渲染、截图、抽样检查，甚至把检查动作写进脚本。
这类任务的难点不在“能不能生成”，而在“生成结果是否真的可用”。

我通常这样判断自由度：如果错了只是表达差一点，可以给 Agent 多一点空间；如果错了会产生假 API、污染代码、破坏构建、交付不可见问题，就要把路径收窄。

## 一个 skill 的编写流程

从零写 skill 时，我现在会按下面这个顺序来。

1. 收集典型任务。

不要从“我要写一个很强的 skill”开始。先列用户会怎么问。比如“帮我把这篇 Markdown 转成发布平台格式”“帮我修这个 DSL 编译错误”“帮我生成一个可运行的移动端页面”“帮我把 PDF 里的表格提取出来”。任务越具体，skill 的边界越清楚。

2. 找出最容易猜错的地方。

这一步比写正向能力更重要。写作任务容易错在标题、开头、编号、图示、空泛结论；代码任务容易错在 API 编造、语法迁移、旧版本写法；构建任务容易错在命令参数、环境变量、产物路径；文档任务容易错在布局和渲染结果。

3. 写 description。

description 要先把门开对。它要说明任务对象、触发场景和排除边界。不要写成万能助手，也不要把“什么时候使用”藏在正文里。

4. 写 SKILL.md 主流程。

主流程要短。Agent 读完以后应该知道第一步要做什么，什么时候读 reference，什么时候跑 script，输出前检查什么。

5. 拆 reference。

把长规则、API 说明、风格画像、错误清单、示例迁移到 `references/`。主文件只保留读取条件。reference 太长时，最好在开头加目录或搜索提示。

6. 脚本化脆弱动作。

如果某个步骤每次都需要搜索、解析、构建、转换、校验，就不要让 Agent 每次临场写。写成脚本以后，skill 会稳定很多。

7. 用真实任务迭代。

第一次写完的 skill 通常只是初版。真正有效的规则，往往来自后续失败：图片文字被裁剪、构建命令猜错、API 被编造、输出格式不适合粘贴、文章句式太模板化。每次返工都应该问一句：这是一次偶发错误，还是应该沉淀进 skill 的新规则。

## 几个很容易写歪的地方

第一个错误是把 skill 写成超长 prompt。长不代表强。真正强的是触发准确、流程短、关键事实不丢、错误路径被拦住。

第二个错误是只写愿望。比如“请生成高质量 UI”“请保持技术深度”“请写可编译代码”。这些话太空。更有效的是写可检查行为：生成 UI 前查组件 API；文章使用图示时必须渲染检查；构建失败先区分编译器阶段和下游构建阶段。

第三个错误是把多个任务塞进一个 skill。写作、代码生成、构建验证、格式转换可能共享背景，但它们的失败模式完全不同。范围太大以后，description 会变模糊，正文会变长，误触发也会变多。

第四个错误是不验证。写作 skill 至少要用真实文章复核；绘图规则要看实际渲染；代码 skill 要查源码和跑构建；文档 skill 要打开或截图检查。没有验证动作的 skill，很容易变成一套看起来正确的愿望。

## 判断一个 skill 是否写好了

我现在会用几个问题快速判断：

- 只看 `name` 和 `description`，Agent 能不能知道什么时候应该用它？
- 触发后读完 `SKILL.md`，Agent 能不能知道第一步要做什么？
- 它有没有明确拦住最常见的错误路径？
- 大块资料有没有拆到 reference，并且主文件写清楚什么时候读？
- 重复、脆弱、可机械化的步骤有没有脚本化？
- 输出前有没有可执行的验证动作？
- 使用几次以后，失败经验有没有回流到 skill 里？

这些问题能回答清楚，skill 就开始脱离普通提示词了。它会变成一套 Agent 可以按需加载的工作流。

Skill 真正有价值的地方也在这里：把个人经验、项目约束、失败案例、检查流程和工具调用方式，压缩成 AI 可以执行的上下文。对复杂任务来说，这比多写几句“请认真思考”有效得多。
